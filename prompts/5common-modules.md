이 프로젝트에 대해 수행해야할 공통 모듈 작업 계획을 작성하라.
페이지 단위 개발을 시작하기 전에 많이 사용될 공통 로직이나 라이브러리 관련 세팅을 마치기 위함이다.
다음과 같은 순서로 진행하라.

1. `/docs/`경로의 문서들을 읽고 프로젝트의 기획을 구체적으로 파악한다.
2. 코드베이스 현 상태를 자세히 파악하고 분석한다.
3.  프로젝트의 기술 스택(언어, 프레임워크 등)을 고려하여, 단위 테스트, 통합 테스트, E2E(End-to-End) 테스트를 포괄하는 견고한 테스트 환경 구축을 계획해라.
 - `docs/rules/tdd.md`의 규칙을 철저히 준수하여 TDD 설계 원칙을 반영
 - 주로 순수 함수(Pure Functions)에 초점을 맞춰 테스트를 설계하고, 외부 의존성이 없는 독립적인 테스트 케이스를 작성하는 전략을 제시
 - 불가피한 경우를 제외하고는 모킹(Mocking) 사용을 최소화하는 방안과 그 이유를 설명
 - 아래 프롬프트를 실행해서, 위에서 설계한 테스트 환경 계획에 대한 다각적 피드백을 받고, 이를 바탕으로 계획을 한단계 더 발전시켜라
   ```markdown
    너는 시니어코딩에이전트야. 아래에 제시된 테스트 환경 구축 계획을 검토하고, 다음 질문들에 대해 구체적이고 실행 가능한 피드백을 제공해 줘.
    1.  **현실성 및 효율성:** 이 계획이 실제 프로젝트 환경에서 효율적으로 작동할 수 있을까? 잠재적인 병목 현상이나 불필한 복잡성은 없는지 평가해 줘.
    2.  **테스트 커버리지:** 계획된 테스트 전략이 핵심 비즈니스 로직을 충분히 검증할 수 있을까? 놓치고 있는 중요한 테스트 시나리오는 없는지 알려줘.
    3.  **유지보수성:** 이 테스트 코드는 장기적으로 유지보수하기 쉬운 구조일까? 가독성, 재사용성, 확장성 측면에서 개선할 점이 있다면 제안해 줘.
    4.  **대안 기술 및 도구:** 현재 제안된 도구(라이브러리, 프레임워크) 외에 더 나은 대안이 있다면, 그 이유와 함께 추천해 줘.
    [여기에 위에서 설계한 테스트 환경 계획 삽입]
   ```
4. 프로젝트 전반에서 공통으로 사용될 핵심 모듈과 로직을 식별하고, 각 모듈의 기능, 책임(역할), 그리고 다른 모듈과의 상호작용 방식을 명확하게 설계한다. 절대 오버엔지니어링 하지않고,설계된 모든 모듈은 반드시 문서에 근거해야 한다. 이 설계 문서를 `/docs/common-modules.md` 경로에 작성한다.
 - 이후 페이지 단위 개발은 모두 병렬로 진행할 수 있도록, 코드 conflict가 생길만한 공통 작성 모듈은 모두 이 문서에 포함되어야만한다. 위 문장은 굉장히 중요하니, 잘 준수되었는지 여부를 3회 이상 반복 확인 및 검증하라.
