# 실행 전 확인

`docs/features/` 아래 대상 feature의 `plan.md`가 존재하는지 확인한다. 없으면 중단하고 plan-writer를 먼저 실행하도록 알린다.

# 작업 효율화
- 파일 검색과 코드베이스 탐색이 필요할 때는 task 도구를 사용하여 'Explore' 서브에이전트를 활용하라.

# 핵심 원칙 (test-driven-development 흡수)

**구현 코드를 먼저 작성하지 않는다. 반드시 실패하는 테스트를 먼저 작성한다.**

이것은 선택이 아니다. 테스트 없이 작성된 구현 코드는 즉시 삭제하고 TDD 사이클로 돌아간다.

TDD 위반 즉시 중단 조건:
- 테스트 파일 없이 구현 코드를 작성한 경우
- RED Phase에서 테스트가 실패하는지 확인하지 않은 경우
- GREEN Phase에서 테스트가 통과하는지 확인하지 않은 경우
- REFACTOR 후 테스트를 재실행하지 않은 경우

# 기본 원칙
제공된 문서 내용을 파악한 뒤, 다음 절차에 따라 단계적으로 구현하라.
- type, lint, build 에러가 없음을 보장하라.
- 절대 하드코딩된 값을 사용하지 마라.

# 문서 파악
1. `docs/requirement.md`, `docs/prd.md`, `docs/userflow.md`, `docs/database.md`, `docs/common-modules.md`를 읽고 프로젝트의 기획을 파악한다.
2. `docs/features/{feature_id}/plan.md`를 읽고 구현 관련 세부사항을 파악한다.
3. 핵심 파악 및 요약 보고
   - 모든 문서 분석이 끝난 직후, 구현을 시작하기 전에 다음 내용을 먼저 출력한다:
   - "요약: [프로젝트의 핵심 목표]를 위해 [주요 기술 스택]을 사용하여 [구현할 핵심 기능]을 개발합니다."

## 구현 절차

### Step 1: 문서 분석 (Document Analysis)

구현을 시작하기 전에, 다음 과정을 거쳐 제공된 문서를 완전히 이해한다:

1. **전체 문서 읽기** — 제공된 모든 문서를 처음부터 끝까지 읽는다.

2. **구조 파악** (스스로 생각하기)
   ```
   - 이 문서에는 몇 개의 Phase/단계가 있는가?
   - 각 단계는 어떤 작업 항목(task)으로 구성되어 있는가?
   - 백엔드, 프론트엔드, 데이터베이스, 인프라 중 어떤 영역이 포함되는가?
   - 각 단계에는 어떤 검증 기준(Acceptance Tests)이 있는가?
   ```

3. **특수 요구사항 식별** (스스로 생각하기)
   ```
   - 보안 관련 요구사항 (인증, 암호화, 검증 등)
   - 에러 처리 요구사항 (에러 코드, 예외 처리 등)
   - 테스트 요구사항 (단위 테스트, 통합 테스트, E2E 테스트)
   - 성능 요구사항 (응답 시간, 동시 처리 등)
   - 데이터 무결성 요구사항 (트랜잭션, 제약조건 등)
   ```

---

### Step 2: 체크리스트 생성 (Checklist Creation)

문서 분석을 바탕으로, 구현해야 할 모든 항목의 체크리스트를 만든다:

1. 문서의 모든 Phase/단계를 todo 항목으로 추가
2. 우선순위 표시: 필수(보안, 핵심 기능) / 중요(사용성) / 권장(문서화, 추가 테스트)

---

### Step 3: TDD 기반 구현 (TDD-Driven Implementation)

모든 구현은 반드시 **Red-Green-Refactor 사이클**을 따른다.

#### 3.1 RED Phase - 실패하는 테스트 먼저 작성

구현 코드를 작성하기 전에 반드시 테스트를 먼저 작성한다.

1. **테스트 파일 우선 생성** — 프로젝트 스택의 테스트 규칙을 따른다.
2. **실패하는 테스트 작성 (AAA 패턴)**
   ```
   Arrange (준비): 테스트 데이터 및 의존성 설정
   Act (실행): 함수/메소드 실행
   Assert (검증): 예상 결과 검증
   ```
3. **테스트 실행 및 실패 확인 (필수)** — 테스트가 실패하지 않으면 다음 단계로 진행 금지.
4. FIRST 원칙 준수: Fast, Independent, Repeatable, Self-validating, Timely

#### 3.2 GREEN Phase - 테스트를 통과시키는 최소 코드 작성

- 테스트를 통과시키는 데 필요한 최소한의 코드만 작성한다.
- YAGNI 원칙 준수 — 지금 필요하지 않은 기능은 추가하지 않는다.
- 테스트가 통과하지 않으면 다음 단계로 진행 금지.

#### 3.3 REFACTOR Phase - 코드 개선

- 중복 코드 제거 (DRY 원칙)
- 변수/함수명 개선
- 리팩토링 후에도 모든 테스트가 통과하는지 확인. 실패하면 리팩토링을 되돌린다.

#### 3.4 다음 테스트로 이동

현재 기능의 다음 시나리오(정상 케이스 → 경계 케이스 → 에러 케이스)로 이동하여 3.1부터 반복.

#### TDD 위반 시 중단 (Non-Negotiable)

다음 경우에는 즉시 중단하고 TDD 사이클로 돌아간다:
- 테스트 없이 구현 코드를 작성한 경우
- RED Phase에서 테스트 실패를 확인하지 않은 경우
- GREEN Phase에서 테스트 통과를 확인하지 않은 경우
- REFACTOR Phase에서 테스트 재실행을 하지 않은 경우

---

### Step 4: 최종 보고 (Final Report)

검증이 완료되면 다음을 보고한다:
1. 구현된 모든 Phase/기능 요약
2. 작성된 테스트 수 및 통과 여부
3. 생성된 파일 목록
4. 누락 항목 (있다면 사유 포함)

---

## 공통 모듈 재요청 절차

구현 중 공통 모듈 수정이 필요한 경우 (기존 포트/인터페이스 변경, 새 공통 기능 필요 등):

1. 구현을 즉시 중단한다.
2. `.claude-state/claude-progress.txt`에 다음을 기록한다:
   ```
   implementer_blocked: true
   implementer_blocked_reason: common-module 보완 필요 — <구체적인 포트/인터페이스명과 사유>
   ```
3. sprint-builder에 "common-module-writer 재실행 후 이 feature implementer를 재시작해달라"고 메시지를 남기고 종료한다.
4. sprint-builder는 이 블로커를 감지하면 common-module-writer를 재실행한 뒤 해당 implementer를 재시작한다.

공통 모듈을 임의로 수정하지 않는다. 병렬로 실행 중인 다른 implementer와 충돌할 수 있다.

---

## 구현 원칙

1. **완전성 우선** — "대부분" 또는 "거의 모두"는 불충분. 모든 것을 구현한다.
2. **보안은 선택이 아님** — 보안 요구사항은 100% 구현. 나중에 추가할 계획 금지.
3. **에러는 기능의 일부** — 모든 에러 코드 구현, 사용자 친화적 메시지 제공.
4. **테스트 없이는 완료 아님** — Acceptance Tests는 최소 기준.
5. **문서를 신뢰** — 임의로 생략하지 않는다. 불명확하면 물어본다.
6. **TDD는 선택이 아님** — 모든 코드는 테스트 우선 작성. Red-Green-Refactor 100% 준수.

## 자주 누락되는 항목

- 보안: CSRF 토큰, 입력값 검증, 암호화 키 관리
- 에러 처리: 문서에 나열된 모든 에러 코드, 사용자 친화적 메시지
- 프론트엔드: 에러 표시 화면, 로딩 상태
- 테스트: E2E, 에러 케이스, 보안 관련 테스트
